# Research / options analyses

Exploratory design-space write-ups that inform the AX.25 roadmap (`../plan.md`) but are **not** commitments — they capture investigation + options + a recommendation at a point in time, for later decision. Each is dated and grounded against the live `packet-net/ax25sdl`, `packet-net/packet.net`, and `packet-net/ax25-ts` trees plus the AX.25 v2.2 spec.

| Doc | Question | Headline |
|---|---|---|
| [pico-packet-node.md](pico-packet-node.md) | Can a lightweight but fully v2.2-compliant packet node run on Pi Pico W-class hardware (RP2040), reusing this stack? | Feasible — Rust + Embassy (`no_std`) consuming the generated tables; AXUDP-over-WiFi tier first; "fully compliant" + "fits 264 KB" reconciled by bounding N1/k via XID. The work is the *runtime*, not the tables. |
| [codegen-reach.md](codegen-reach.md) | Beyond the SDL state-machine tables, what else is spec-derived enough to codegen across languages (shrinking the hand-written per-language runtime)? | Generate **cross-language conformance vectors first** (drift-proofs the un-generatable majority); then the SP-010 guard/event closed sets; then the frame + XID wire codecs. |
| [pico-w-rust-dev-workflow.md](pico-w-rust-dev-workflow.md) | What's the dev *workflow* for a Rust RP2040 / Pico W node — and can an autonomous agent flash/run/test hands-free? | The portable `no_std` logic is **fully hands-free on the host** (`cargo test`, zero hardware) — that's the bulk of the work. The WiFi tier needs a **real Pico W** (no emulator does CYW43 — Wokwi/rp2040js/Renode all WiFi-blind), but a board left wired to an SWD probe + `probe-rs run` on a self-hosted runner *is* a real hands-free loop. Embassy + `cyw43` + `embassy-net`; `embedded-test` on-target. |
| [headless-soundmodem.md](headless-soundmodem.md) | Can PDN grow a native headless (no Qt) C# soundcard modem — QtSoundModem as reference, DCD through the `ICarrierSense` seam, browser waterfall — and what would it take? | Feasible; the seams already exist and managed DSP is ~14 % of one old x64 core for the worst case (measured). Gates: a QtSM port must be **GPL-3.0-or-later** (forces resolving the repo's MIT/AGPL contradiction), and the live network is 100 % NinoTNC IL2P+CRC / 0 % 1200 AFSK — so NinoTNC-waveform interop, not QtSM parity, is the bar. (2026-07-14) |

The first two were produced during the v2.2 completion arc (2026-06); the codegen-reach analysis is the companion to the Pico one ("the work is the runtime, not the tables"). The dev-workflow note operationalises the Pico one's phased path: it answers *how* an agent would iterate (host vs emulated vs real board) and which parts are hands-free today.
